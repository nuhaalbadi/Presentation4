› Cognitive Walkthroughs

The cognitive walkthrough is a usability inspection method that does not require users and can be done in early UI development stages. The cognitive walkthrough requires having a set of tasks that the UI support, description of the UI or mockup, and intended users background. Once all these three elements are available, UI evaluators/designers “walkthrough” each task while reasoning whether or not each step required to perform the task can be easily recognized and done by the user.

The cognitive walkthrough was developed by Clayton Lewis, Peter Polson, Cathleen Wharton, and John Rieman in 1992. If you would like to learn about the theory behind the methodology, you can do so by reading Cognitive Walkthroughs: A Method for Theory-Based Evaluation of User Interfaces by Ploson et al.

  • Guidelines

    • The cognitive walkthrough is designed to predict usability issues that users might encounter when they first interact with the UI, i.e. without training.

    • The cognitive walkthrough should be done before implementing the UI, as Lewis and Rieman state, “the walkthrough is really a tool for developing the interface, not for validating it.”

    • the cognitive walkthrough can be done effectively on a detailed written description of a UI as it would on a working prototype. In fact, when the cognitive walkthrough is done on a written description of a UI, it can detect flaws in the UI documentation such as missing description of some UI components.

  • Before doing the walkthrough

    Before performing a cognitive walkthrough, you need to have the following:

    • A detailed description of the UI or a prototype. The prototype does not need to be computerized, it can be a paper-based prototype.

    • Well defined tasks that the UI support. You do not need to do a cognitive walkthrough for every task that the system support. However, it is important to carefully select the tasks that will be tested in a way that cover almost every feature of the UI.

    • Well defined sequence of steps (actions) for each task that are needed to complete the task using the UI.

    • A definition of the UI users including their background and previous knowledge or experience on specific areas that could affect the way they use the UI.

  • Walking through the tasks

    After having all the requirements described in the previous section, a group of UI evaluators or designers gather around to tell a convincing story on how users will accomplish each task by walking through each step (action) while reasoning on whether or not it would be obvious for the user to do each step. UI evaluaters should critique the story to make it sound and convincing. Lewis and Rieman suggest asking the following questions while walking through each action.

    1. Will users be trying to produce whatever effect the action has?

    2. Will users see the control (button, menu, switch, etc.) for the action?

    3. Once users find the control, will they recognize that it produces the effect they want?

    4. After the action is taken, will users understand the feedback they get, so they can go on to the next action with confidence?

  • Advantages

    • It does not require actual users to do the cognitive walkthrough, which can save a lot of time.

    • It can be done on early design stages and dose not need special tools to perform the cognitive walkthrough.

    • The cognitive walkthrough is simple and easy to learn and master.

    • It can can best detect usability problems related to first-time users of a system.

  • Disadvantages

    • It might take a long time to perform the cognitive walkthrough on complex tasks.

    • The efficiency of the cognitive walkthrough depends on the judgment of the UI evaluator(s). If his/her judgment is not reliable, then the results could be misleading.

  • Further Reading

    • An Introduction to Usability by Patrick Jordan is a freely available online book that does not only talk about the cognitive walkthrough method, but also gives a brief introduction to many other useful methods.

    • Task-Centered User Interface Design by Lewis and Rieman (1993) sec 4.1 describes in detail the cognitive walkthrough and gives useful examples on how to conduct the cognitive walkthrough effectively.
